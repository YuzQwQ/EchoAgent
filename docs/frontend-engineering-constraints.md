# Frontend 工程约束清单（防屎山版）

> 目标：你要求的第 3 件事——“防止再次回到超高耦合屎山”的约束规则。

## 1. 文件与模块约束

- 单个业务文件建议不超过 **300 行**，超过必须拆分。
- `index.html` 只保留页面骨架与入口，不承载核心业务。
- 模块职责单一：
  - 通信
  - 状态
  - 视图
  - 设备能力
  - 编排

---

## 2. 状态管理约束

- 全局状态只允许存在于 `settings-store` / `app-store`。
- 禁止在多个模块中重复维护同一状态（如 `isConnected`）。
- 持久化键统一前缀：`echo.*`，避免冲突。

---

## 3. 事件与调用约束

- 模块之间优先通过事件总线通信，不直接互相调用内部方法。
- 每个事件都必须有 schema（至少在注释中声明字段）。
- 禁止“字符串协议”散落：统一常量定义。

---

## 4. UI 与逻辑分离约束

- DOM 查询统一放在 `view` 层，不允许业务模块随处 `getElementById`。
- 样式与行为分离：禁止在业务代码里拼大量 inline style。
- 弹窗、提示、错误展示走统一 `ui-notify` 模块。

---

## 5. Electron 兼容约束

- 所有 Electron API 必须先做 capability 检测。
- Browser 降级路径必须有明确提示，不可静默失败。
- IPC 事件监听必须集中注册、集中注销。

---

## 6. 网络与稳定性约束

- WebSocket 必须有：
  - 重连策略
  - 最大重试/退避
  - 连接状态机（connecting/open/closing/closed）
- 所有请求必须有超时与错误分类。

---

## 7. 日志与调试约束

- 日志接口统一（`logger.info/warn/error`）。
- 生产环境可降级日志级别。
- 每个关键链路带 traceId：
  - 用户输入
  - ws chunk
  - 音频序列

---

## 8. PR 质量门禁（推荐）

每个前端 PR 必须满足：

- [ ] 改动描述包含“模块边界变化”
- [ ] 至少 1 条 smoke test 记录
- [ ] 无新增超长函数（>80 行）
- [ ] 无新增跨模块循环依赖
- [ ] Browser / Electron 兼容结果说明

---

## 9. 重构策略（避免一次性爆炸）

- 采用“**剥洋葱式重构**”：
  1. 抽接口
  2. 平移逻辑
  3. 替换调用
  4. 删除旧实现
- 每次只处理一个子域（设置、通信、字幕、观察、音频）

---

## 10. Definition of Done（前端）

满足以下才算完成：

1. 功能可用（文本/设置/连接）
2. 拆分后职责清晰
3. 关键路径有日志可追踪
4. Browser 与 Electron 行为可预期
5. 文档已更新
